# Инструкция для Claude — работа с сайтом adorelanguages.com

Этот файл — памятка на случай, если новая сессия Claude ничего не помнит о предыдущей работе.
Прочитай его целиком перед тем, как что-то менять в репозитории. Если что-то в реальном коде
разошлось с этим описанием — доверяй коду, а не файлу, и обнови файл заодно.

## Что это за проект

Статический сайт (GitHub Pages, кастомный домен через CNAME → adorelanguages.com), блог
"Обожаю Языки" — посты про этимологию, идиомы, бренды и лингвистические факты. У Вадима также
есть тот же контент в Facebook (fb) и на сайте (site) — это два параллельных каталога.

Ключевая особенность блога: **весь содержательный текст — на картинках** (посты — это готовые
карточки с текстом, нарисованные заранее). На страницах сайта НЕТ и не должно быть видимого
текста-дублёра — только картинка, теги (скрыты) и мета-данные для SEO. Исключение: у некоторых
старых постов есть собственные видимые комментарии автора под картинкой (`<ul class="bullets">`) —
их не трогать и не добавлять новые по своей инициативе.

## Структура репозитория

- `index.html` — главная страница сайта (adorelanguages.com), с большой картинкой собаки,
  круглой кнопкой «Поиск» и ссылками на ФБ/сайт/Instagram/Telegram.
- `fb/<slug>/index.html` — просто редирект (`meta http-equiv="refresh"`) на конкретный пост в Facebook.
- `site/<slug>/index.html` — полноценная страница поста: картинка, мета-теги, JSON-LD, блок разделов, теги.
- `images/posts/<slug>.png` — картинка поста (обычно 756×756).
- `links/fb/posts.json` — каталог постов для ФБ (slug, title, tags).
- `links/site/posts.json` — тот же список + поле `"bg"` (см. ниже про цвета).
- `links/site/sections.json` — какие посты входят в какие разделы.
- `links/site/section/<key>/index.html` — страница одного раздела.
- `links/site/all/index.html`, `links/fb/all/index.html` — общая лента всех постов.
- `links/site/index.html` — страница поиска по тегам/названию + список всех разделов.
- `scripts/generate_sitemap.py`, `scripts/ping_indexnow.py` — запускаются GitHub Action'ом
  (`.github/workflows/update-sitemap.yml`) автоматически при пуше в `links/site/posts.json` или
  `links/fb/posts.json` — **sitemap.xml обновлять руками не нужно**, бот сам закоммитит.

## Как добавить новый пост — пошагово

Вадим обычно присылает: заголовок, раздел(ы) (или «без раздела»), ссылку на пост в Facebook,
список тегов через `#`. Картинка обычно уже лежит в подключённой папке `Загрузки/посты` —
проверяй её первым делом (Glob по этой папке), имя файла = точное название поста + `.png`.

1. **Прочитать картинку** (инструмент Read) — это единственный источник текста поста.
2. **Определить bg** — открыть картинку через PIL, взять цвет пикселя `(5,5)`:
   - `pink`  ≈ rgb(245, 220, 207)
   - `lilac` ≈ rgb(233, 206, 242)
   - `blue`  ≈ rgb(70, 19, 89)   ← это тёмно-фиолетовый на вид, но в системе Вадима это «синий»!
   - `green` ≈ rgb(240, 247, 205)
   Bg обязан быть одним из этих четырёх значений — это bucket для JS-шаффла на страницах каталога
   (см. `links/site/all/index.html` — цикл pink → lilac → blue → green, все посты внутри бакета
   перемешиваются случайно при каждой загрузке страницы, порядок цветов не меняется никогда).
3. **Сгенерировать slug** — транслитерация заголовка, без пробелов и разделителей, всё слитно
   строчными буквами (например «Диванная картошка» → `divannayakartoshka`, «Цветная собака» →
   `tsvetnayasobaka`). Если Вадим сам не даёт slug — генерируй сам по этому правилу.
4. **Проверить дубликаты** — прогнать по `links/fb/posts.json` и `links/site/posts.json`, что
   такого slug/title ещё нет.
5. **Написать тексты**:
   - Полное описание (для `meta name="description"` и JSON-LD `"description"`, должны совпадать
     дословно) — переписать всё содержание карточки в связный текст, обычно начиная с
     «НАЗВАНИЕ ЗАГЛАВНЫМИ. Первая фраза...».
   - Короткая версия (для `og:description` и `twitter:description`, они между собой совпадают, но
     ОТЛИЧАЮТСЯ от полного описания) — один цепляющий заголовок-крючок в одно предложение.
   - Alt-текст картинки — «Название — краткая суть поста».
6. **JSON-LD keywords** = теги ровно как дал Вадим (ничего не добавлять от себя!) + всегда
   фиксированный хвост: `, обожаюязыки, обожаюязыкознание, языкознание, лингвистика, переводчики,
   иностранныеязыки, полиглот, филология, наукаоязыке`.
7. **Добавить записи** в конец массивов `links/fb/posts.json` (slug, title, tags) и
   `links/site/posts.json` (slug, title, tags, bg) — редактировать через `json.load`/`json.dump`
   с `indent=2, ensure_ascii=False` (это уже совпадает с текущим форматированием файлов, диффы
   получаются минимальные).
8. **Создать `fb/<slug>/index.html`** — короткий редирект на присланную ссылку Facebook (шаблон
   ниже).
9. **Создать `site/<slug>/index.html`** — по шаблону (ниже). Блок разделов:
   - Если раздел один — `<p class="related-title">Раздел:</p>` и один `<li>`.
   - Если разделов два и больше — `<p class="related-title">Разделы:</p>` и по `<li>` на каждый.
   - Если без раздела — весь `<!--SECTIONS--> ... </div>` блок просто не добавлять (сразу от
     картинки к `.toggle`).
   - Для идиом с языком — ссылка вида `/links/site/section/idiomy/?lang=<язык>`, текст
     «Идиомы: <язык>».
10. **Обновить `links/site/sections.json`**, если пост относится к разделу(ам) — добавить slug в
    нужный `"slugs"` массив (или в `idiomy.languages.<язык>`). Редактировать точечной заменой
    строки (не пересобирать весь JSON), там компактный однострочный формат массивов.
11. **Скопировать картинку** в `images/posts/<slug>.png`.
12. **Закоммитить и запушить** (см. раздел про git ниже).

Если нужно создать **новый раздел** — добавить ключ в `sections.json`, скопировать
`links/site/section/strany/index.html` как шаблон (это «простой» раздел без языкового фильтра) и
заменить 4 вхождения: `<title>`, `<h1 id="title">`, `sections['strany']` → `sections['ключ']`,
и строку `` `Страны (${posts.length})` ``. Добавить ссылку на раздел в `links/site/index.html` в
блок `.sections`, подобрав следующий цвет по кругу: `#F4DBCE` (персиковый) → `#E9CEF2` (сиреневый)
→ `#461359` с `color:rgb(129,239,135)` (тёмный, светло-зелёный текст) → `#F0F7CD` (светло-зелёный)
→ по кругу.

## Шаблон `fb/<slug>/index.html`

```html
<!DOCTYPE html>
<html>
<head>
  <meta charset="utf-8">
  <meta http-equiv="refresh" content="0; url=ССЫЛКА_НА_FACEBOOK">
</head>
<body>
  <p>Redirecting to Facebook…</p>
</body>
</html>
```

## Шаблон `site/<slug>/index.html`

Проще всего скопировать структуру любого недавнего поста (например `site/morgenmuffel/index.html`
или `site/trampova/index.html`) и заменить: title, meta description, og/twitter title+description,
JSON-LD (headline/description/keywords/url), путь к картинке, alt, блок `.related` (или его
отсутствие), список тегов в `.tags`. CSS-блок в `<style>` у всех "простых" постов одинаковый —
копировать один в один.

## Правила, которые легко забыть

- **Никогда не добавлять видимый текст на страницы постов** — это специально нарушает концепцию
  блога (текст только на картинках). Скрытый на 100% текст (`display:none` без возможности
  раскрыть) тоже решили не делать — обсуждали риски с точки зрения политик Google по скрытому
  контенту, да и не решает реальную причину непроиндексированных страниц (см. ниже). Единственный
  согласованный инструмент SEO — meta description + JSON-LD + короткое summary для og/twitter.
- Теги в `.tags` и в `posts.json` — **ровно то, что прислал Вадим**, без отсебятины. Единственное
  дополнение, которое можно делать автоматически — фиксированный хвост в JSON-LD keywords (см. п.6).
- Раздел/Разделы — строго по числу `<li>` (проверять после любого добавления второго раздела
  существующему посту).
- bg — только 4 строго фиксированных значения, определять по реальному цвету пикселя, а не на глаз.
- Слаг — транслитерация без разделителей, слитно.
- **Фарси / персидский — это один и тот же язык под двумя названиями.** Если пост относится к
  этому языку (в разделе «Языки» и/или «Идиомы»), всегда добавлять ОБА ключа — `"фарси": [...]`
  и `"персидский": [...]` — в соответствующий `languages` объект в `sections.json`, оба со
  slug'ом этого поста. В блоке `.related` на странице поста писать одну строку на раздел вида
  «Языки: фарси / персидский» (или «Идиомы: фарси / персидский»), ссылка при этом ведёт на
  `?lang=фарси` (неважно на какой из двух — раздел покажет тот же список постов). Если для
  какого-то из двух разделов есть только одна из привязок — это баг, найденный и исправленный один
  раз (пост `lyuboviorgany`) — проверять оба раздела сразу, когда добавляешь этот язык.
- **Тот же приём для других языков с альтернативными названиями** — например адыгский / черкесский
  (пост `kak-dela`). Всегда добавлять оба ключа-названия в `languages` с одним и тем же slug'ом,
  а в `.related` писать одну строку вида «Языки: адыгский / черкесский», ссылка на `?lang=<любое
  из двух>`. Список таких known-пар языков: фарси/персидский, адыгский/черкесский — если появится
  ещё одна пара синонимичных названий, добавлять её сюда же.
- Перед пушем — `git pull`, потому что бот `sitemap-bot` может успеть закоммитить
  `sitemap.xml`/`.indexnow_state.json` между сессиями.

## Git / деплой

- Репозиторий: `github.com/adorelanguages/adorelanguages`, ветка `main`, деплой — GitHub Pages
  (классический branch-deploy, без своего workflow для Pages).
- Клонировать в `/tmp` (не в примонтированную `outputs`-папку — там git ломается на спец-файлах
  вроде `.git/config.lock`, монтированная файловая система их не поддерживает).
- Для пуша нужен GitHub Personal Access Token от Вадима (он присылает его в чат по запросу).
  Использовать так, не логируя токен в вывод:
  ```
  git push "https://x-access-token:${GITHUB_TOKEN}@github.com/adorelanguages/adorelanguages.git" HEAD:main
  ```
- Если токен уже был один раз использован и отправлен в чат открытым текстом — стоит посоветовать
  Вадиму отозвать/перевыпустить его после завершения задачи (это разовая гигиена, не запрет).
- Картинки постов Вадим кладёт в подключённую папку `Загрузки/посты` (Downloads/посты) на его
  компьютере — она видна через Read/Glob по пути `C:\Users\vadim\Downloads\посты`, а в bash-песочнице
  как `/sessions/.../mnt/посты/`.

## Разное, что уже обсуждали и решили

- **Cowork не хранит память между обычными сессиями** (это открыто признанное ограничение beta на
  момент этого разговора) — только внутри Project. Этот файл — способ обойти это ограничение для
  работы над сайтом конкретно.
- **SEO / индексация в Google**: главная причина непроиндексированных страниц у Вадима —
  «Discovered — currently not indexed» (Google ещё не дошёл до крауления, это про приоритет для
  нового домена, а не про качество контента) — «Crawled — currently not indexed» (реально про
  тонкий контент) была у него лишь на паре страниц. Хэштеги/тонкий контент — не главная проблема,
  решать её добавлением видимого текста не будем (см. правила выше).
- **Мобильная адаптивность**: если Вадим присылает скриншоты «сайт выглядит по-разному на разных
  айфонах» — сначала проверь, не связано ли это с настройками конкретного устройства (Safari
  Page Zoom для сайта, или Настройки → Экран и яркость → Вид → Стандарт/Увеличенный), а не сразу
  чини вёрстку — в прошлый раз разница объяснялась именно этим, а не багом кода.
- Стандартные цвета интерфейса сайта: основной текст/навигация `rgb(45, 44, 91)`, шрифт — кастомный
  `Factora` (файлы `FACTORA-*.OTF` в корне репозитория).
- **Тире**: Вадим не любит длинное тире (—), даже там, где оно стандартно. Везде используем короткое
  тире (–), не дефис (-). Это касается и текста постов, и служебных сообщений на страницах (например,
  пустых состояний на страницах голосований).
- **Страницы голосований** (`/golosovania/<slug>/` и старая `/idei/`): пустое состояние списка вариантов
  должно звучать как «Пока нет одобренных вариантов – стань первым, кто предложит свой!» — короткое
  тире, «стань» (не «станьте»). Соблюдать в любых новых страницах голосований.
